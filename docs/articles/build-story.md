这个站你现在正在看。它是我断断续续搭起来的，前后拖了挺长一段时间。主要不是因为难，而是我用的是 Claude Pro，每天的额度就那么多，推进节奏基本被额度卡着——写一阵、把额度用光，就得停下来等第二天。所以这不是那种「一个周末做完」的项目，是攒着零碎时间一点点磨出来的。它是一个博客前台，加一个藏在登录后面的私有网盘。后端 Python，数据库就一个 SQLite 文件，存储挂在七牛云上。全程只有一台四核四 G 的轻量服务器，和一个 AI 助手陪我敲。

这篇文章是回头看这段过程。不是教程，我不会一步步教你 `npm create`。我想讲的是那些当时纠结了很久的选择：为什么是这个技术、踩了哪个坑、现在回头看值不值。有些决定我到现在都觉得赚了，有些我中途推翻重来过一次。

## 为什么不是 Hexo，不是 WordPress

我一开始就没打算用现成的博客系统。

Hexo、Hugo 这类静态生成器，写博客够了，但我要的不只是博客。我要一个能登录进去、上传文件、在线预览图片和文本的网盘，而且这个网盘后端得是纯 API，因为我以后可能想用别的客户端连它，比如直接挂到 Finder 里。静态生成器给不了这个。

WordPress 能给，但它太重了。一个 PHP + MySQL 的庞然大物，插件生态是它的优点也是它的负担，我不想为了写几篇文章去维护一套我并不完全理解的系统。

所以是自己写。技术栈定得很快：

后端 FastAPI。Python 写起来简单，自带 OpenAPI 文档，这点对我很重要，因为「以后可能有别的客户端」意味着接口得清晰、能自解释。数据库 SQLite，不是因为它先进，而是因为它是一个文件。备份就是拷贝那个文件，没有单独的数据库进程要守护。对一台四 G 的机器来说，少一个常驻进程就是省一份内存。

前端 Vue 3 加 Vite。理由也简单：构建快，上手快，和「简洁干净」这个调性合得来。

代价我当时也认了：SQLite 我没上任何迁移工具（Alembic 之类）。表结构直接靠 `create_all` 建。这意味着以后改模型字段，我得手动删库重建。对个人项目、数据量小，这个代价可以接受，但如果哪天数据重要到不能丢，这个决定就得改。这是那种「现在省事、将来还债」的典型取舍，我知道我在借什么。

## 地基：一个人的后台，不需要注册系统

后台要登录。但整个系统只有我一个用户。

这个前提简化了非常多东西。我不需要注册流程，不需要找回密码，不需要权限分级，不需要防刷注册的验证码。我只需要一个脚本 `seed_admin.py`，启动时往库里塞一个管理员账号，密码用 bcrypt 哈希存着。登录换一个 JWT，前端每次请求带上，后端验签。就这些。

省掉一整套用户系统，不是偷懒，是把复杂度花在真正需要的地方。一个人用的东西，搞多用户权限是自己给自己找麻烦。

鉴权我用 PyJWT 加 bcrypt，没用更常见的 python-jose 和 passlib。原因很实际：前两个维护更活跃、依赖更少。依赖少这件事，在一台你要自己运维、还得用国内镜像装包的服务器上，是实实在在的好处。每多一个依赖，就多一个哪天装不上或者出安全公告的可能。

## 文件管理：这个项目最重的一块

如果说博客部分是四平八稳，那网盘就是这个项目真正难的地方。它我推翻重写过一次。

最开始我图快，用了一个现成的文件管理器组件 VueFinder。它能跑，列目录、预览、上传都有。但用了两天我就发现，它的交互不是我想要的样子，而且我要在它之上做的定制（缩略图、特定的上传流程）总是别扭。于是有一天我把它整个下掉了，自己用 Element Plus 加七牛的 `qiniu-js` 从头写了一个。目录树、面包屑、右键菜单、框选多选、拖拽上传、拖动移动，全部自己实现。工作量翻倍，但每一处交互都是我想要的。

这个决定值不值？值。现成组件适合验证想法，但一旦你对细节有强烈意见，魔改别人的东西往往比自己写还累。

难点不在 UI，在存储模型。文件本体存在七牛云的私有空间里，本地 SQLite 只存元数据。这里有几个关键设计，每一个背后都是一次踩坑或者一次想清楚。

**上传是前端直传七牛，不经过我的后端。** 一个文件如果要先传到我服务器、再由服务器转手传七牛，那我这台小机器的带宽就成了瓶颈，而且白白过一遍流量。所以流程是这样的：

```viz
upload-flow
```

前端要传文件时，先向后端要一张「上传凭证」。这张凭证是后端用七牛的 SecretKey 现签的，但关键在这：它被限定到一个具体的文件 key、而且短时效。SecretKey 本身永远不出后端。前端拿着这张一次性凭证，用 `qiniu-js` 直接把字节传给七牛（小文件表单直传，大文件自动分片、断点续传）。传完之后，前端再调一次后端的 `/register`，把「这个路径对应七牛的哪个 key」登记进数据库。

登记这一步我做了防御：后端会先去七牛 `stat` 一下，确认这个对象真的落地了，而且以七牛返回的文件大小和类型为准，不轻信前端上报的数字。校验不过就拒绝登记。这样「数据库里有记录」就严格等价于「七牛上真的有这个文件」，不会出现前端说传成功了、实际七牛上没有的幽灵文件。

**改名和移动只改数据库里的路径，不动七牛上的对象。** 七牛的 key 是一串不透明的 uuid，和用户看到的文件名、路径完全解耦。所以你在网盘里把 `a.txt` 改名成 `b.txt`，我只改数据库里那一行的 path 字段，七牛上那个对象纹丝不动。移动到别的文件夹也一样。只有「复制」才会真的去七牛复制一份对象。这个设计让改名、移动这些高频操作变得几乎零成本。

而且因为文件夹也存在同一张表里（用一个 `is_dir` 标记，key 为空），空文件夹也能持久化，这是很多简易网盘做不到的。

**预览和下载不走我的后端，直接 302 跳转到七牛的签名 URL。** 又是为了省流量。你点开一张图，后端只做一件事：验证你登录了，然后回一个 302，把你重定向到一个七牛的私有签名地址。之后是你的浏览器直接去七牛拿字节，和我的服务器再无关系。下载也一样，还能用七牛的 `attname` 参数强制附件名。

这里有个头疼的细节。这两个 GET 请求是浏览器的 `<img>` 标签或者下载链接直接发起的，它们带不上 `Authorization` 头。所以我额外支持了 URL 里带 `?token=`，前端拼地址时把 token 附上去。这种「本该用 header 却只能塞进 query」的妥协，在做前后端分离时会反复遇到。

**图片会先缩略再传给你。** 网格里显示一堆缩略图，如果每张都拉原图，流量和加载速度都难看。七牛有个 `imageView2` 的图片处理能力，可以在 URL 里加参数，让它返回一个缩小、转成 webp、压过质量的版本。网格用小尺寸裁剪铺满，侧栏预览用中等尺寸等比缩放，只有你点开放大查看器时才拉原图。效果有多明显？我实测过一张 416 KB 的 PNG，缩略成 webp 之后只有 2.3 KB。

不过这里埋着一个只有真部署了才会发现的坑：本地开发时，七牛给的测试域名是按路径缓存的、忽略 URL 上的 query 参数，而缩略图的处理指令恰恰全在 query 里。所以本地开发根本看不到缩略效果，图片会回落成原图显示。这个特性只在生产环境、绑定了正式域名之后才生效。我当时对着本地画面调了半天缩略图参数，以为自己写错了，后来才想明白是测试域名的锅。

## 把网盘挂进 Finder：WebDAV 这层翻译

网页管理器做完，我起了个念头：能不能把这个网盘直接挂到 macOS 的访达里，像本地文件夹一样拖拽？

能。协议叫 WebDAV，一个挺老的标准。我写了一层适配，把数据库里那棵 path↔key 的树，翻译成标准的 WebDAV 响应。这样 Finder、rclone、RaiDrive、Mountain Duck 这些客户端都能挂上来浏览、预览、下载，甚至直接编辑。

这一层比想象中麻烦，因为不同客户端的脾气差别很大，几个坑值得记：

**macOS 的访达要求服务器声明支持「锁」，否则只让你只读挂载。** 但我这是单管理员，根本不存在并发写冲突，实现一套真正的锁管理纯属浪费。于是我做了个「假锁」：客户端来请求锁，我总是爽快地授予，回一个锁令牌，但我根本不追踪它。就为了让访达相信这里可以写。

**大文件上传不能把整个文件读进内存。** WebDAV 的上传是 PUT，而且上传的字节必须经过我的后端（这点没办法），因为 WebDAV 客户端只会 PUT 给我，七牛不直接收 WebDAV。如果我天真地 `await request.body()` 把整个文件读进内存再转发，传一个几 G 的文件就能把这台四 G 的机器顶爆。所以我改成边收边写一个临时文件，内存占用恒定，再让七牛 SDK 从磁盘分块读上去。下载可以 302 直连省流量，上传这条路绕不开，只能老实做流式。

**客户端分两拨，取字节的方式得区别对待。** 有些客户端（rclone、Cyberduck、Mountain Duck）会乖乖跟随 302 重定向，那就让它们直连七牛，省我流量。但 macOS 自己的 webdavfs、Windows 的 mini-redirector，跨域 302 不可靠，只能由我的后端代理取字节，还得透传 Range 请求头，因为 QuickLook 预览、流媒体、断点续传都要分段读。我靠 User-Agent 来分流。

最后我把 WebDAV 放到了一个专用子域名 `dav.dawnop.com` 上，让客户端只填一个干净的根地址就能挂。这比塞在主域名的子路径下兼容性好得多，尤其是 iOS 和一些内核级的挂载器，它们会假定 WebDAV 的根就是站点的根。

## 搜索：在四 G 机器上，我不想为它单开一个进程

博客文章多了，得能搜。

搜索引擎的第一反应可能是 Elasticsearch 或者 Meilisearch。但它们都是独立进程，基线内存动辄一百多兆，还要单独守护、开端口、做数据同步。在一台已经跑着后端、Nginx、前端静态资源的四 G 机器上，再塞一个常驻一百兆的搜索服务，我舍不得。

而且我的语料就几十上百篇文章，不是海量数据。独立引擎那些强项——处理 TB 级数据、复杂的分面检索——我根本用不上，却要为它付出内存和运维的代价。

答案是 SQLite 自带的 FTS5 全文索引。它就在数据库进程里，随后端一起起，几乎不额外占内存。文章增删时靠触发器自动同步索引，天然一致，不用写什么双写管道。

真正麻烦的是中文分词。FTS5 自带的 `trigram` 分词器对中文有硬伤：它按连续三个字符切，像「内存」这种两个字的词根本命不中，只能退回慢吞吞的 `LIKE` 扫描。社区常用的方案是 wangfenjin 那个 simple 扩展。我在两条路里犹豫过：

simple 提供了 jieba 词级分词，分词质量高、支持拼音。但我实测发现它常驻要吃大约 200 MB 内存，还要带一个十几兆的词典，对我这台机器又是硬成本。而且我一度以为词级分词能消除「京大」误匹配「北京大学」这种跨词假阳性，后来发现不能，jieba 也一样会。

于是我选了 simple 的另一个模式：纯字级加拼音，不启用 jieba。它几乎不占内存（就是那个九百 KB 的 `.so` 文件），召回导向，能搜中文、能搜拼音、甚至能搜拼音首字母。对个人博客，这个质量完全够用。

我还给分词器做了个逐级降级：启动时优先加载 simple，加载不到就退到内置的 trigram，再不行就退到 LIKE。缺了扩展也不影响服务启动，只是搜索质量降一档。这种「优雅降级」在依赖外部二进制的地方特别重要。说到这个，又是一个只有部署才会撞上的坑：那个 simple 的 `.so` 文件我没法让服务器自己下载。我写的抓取脚本在服务器上连不上 GitHub。最后只能在本地下好 Linux 版的二进制，手动 scp 上去。而且因为本地是 macOS 的 `.dylib`、服务器是 Linux 的 `.so`，每次同步后端代码还得记得把这个文件排除掉，免得把 Mac 的二进制覆盖上去。

前端我把搜索做成了一个 ⌘K 唤起的命令面板，输入即搜，方向键上下选，回车打开。就是现在你按 ⌘K 会弹出来的那个。

## 性能：首页曾经慢得让我自己都嫌弃

有一阵子，这个站的首页第一次加载慢得离谱。我自己打开都要等。

扒下来看，原因是一堆：Element Plus 全量引入，图标也全量注册；`main.js` 顶层就把 highlight.js（全语言包）、KaTeX、Markdown 编辑器一股脑引进来了，可首页根本用不到这些；那个还没下线的 VueFinder 也被塞进了首屏；更别提 Nginx 只压缩了 HTML，JS 和 CSS 裸传，加起来两兆多。

一台小机器，加上国内的网络环境，这么大的首屏就是灾难。我花了一天专门治它：

Element Plus 改成按需引入，用到哪个组件才打包哪个组件的样式，图标也改成按引用引入。那些重依赖——Markdown 编辑器、KaTeX、代码高亮——全部挪到懒加载的路由里，首页不碰它们。highlight.js 从全语言包砍到只保留十六种常用语言。构建时用 gzip 把产物预压缩成 `.gz`，Nginx 开 `gzip_static` 直接发预压缩文件，省掉每次请求现压的开销。再把 Vue、Router、axios 这些几乎不变的框架代码拆成一个单独的 vendor 块，好让浏览器长期缓存。

数字说话：入口 JS 从 1757 KB 压到 121 KB，CSS 从 736 KB 压到 45 KB。首屏 gzip 后大约 120 KB，其中还有五十多 KB 是可以长缓存的 vendor；回访的时候，真正要重新下载的业务代码只有大约 33 KB。

后来备案批下来，我又把静态资源挂上了七牛的 CDN。这是最后一块拼图，下一节讲。

## 上线、备案、和那些运维琐事

本地跑通只是开始，真正的麻烦在上线。

部署形态很朴素：Nginx 监听 80 和 443，托管前端构建出来的静态文件，把 `/api` 反向代理到本地的 uvicorn；uvicorn 由 systemd 守护。没有 Docker（除了后面自建的密码管理器），没有 Kubernetes，一台机器该有的样子。

**HTTPS 证书我用 acme.sh 自动续。** 申请的是 `*.dawnop.com` 的通配符证书，这样主站、WebDAV 子域名、后面的密码管理器子域名都能复用一张。续期用 DNS 验证，通过 DNSPod 的 API 自动加删解析记录。这里有个容易被忘掉的环节：七牛那边的域名也用着同一张证书，证书续期后必须同步上传到七牛、重新绑定。我把这个动作挂进了 acme.sh 的续期钩子里，证书一更新，脚本就自动把新证书推给七牛的两个域名（存储域名和 CDN 域名）改绑。忘了这一步的话，三个月后证书一换，七牛那边的 HTTPS 就悄悄失效了。

**备案。** 国内服务器绕不开的一道手续。ICP 备案批下来之后，工信部要求备案号必须挂在网站底部、链接到它的查询平台；网站上线三十天内还得办公安备案，批下来又要挂一个公安备案号加图标。这两个号我都挂上了。

但备案号怎么在一个开源项目里管理，是个有意思的小问题。我的代码是开源的（在 GitHub 上），我不想把自己的备案号写死在代码里，那是我的个人身份信息，而且别人 fork 这个仓库就不该带着我的备案号。所以我把它做成了构建时的环境变量，放在一个不入库的 `.env.local` 文件里。有配置就渲染，没配置页脚就只剩个版权行，干干净净地退化。别人 clone 下去，得到的是一个空白模板。

**CDN。** 备案一通过，我就把静态资源挂上了七牛 CDN。方案是回源：CDN 上放一个 `cdn.dawnop.com` 的加速域名，回源到我的 Nginx。页面的 HTML 还是源站直出（方便发版即时生效），但页面里引用的 JS、CSS、字体全走 CDN。构建时通过一个环境变量把资源地址前缀指向 CDN，和备案号用的是同一套「不入库」的管理方式。

配 CDN 撞了两个坑值得一提。一是那个 CDN 专用的 Nginx 块不能像主域名那样把 80 跳转到 443，因为跳转的目标是 CDN 自己，会形成回源死循环。二是 `/assets` 目录必须带上跨域响应头，因为页面在主域名、资源在 CDN 域名，浏览器加载跨域的模块脚本和 KaTeX 字体会要求 CORS，而这个头会被 CDN 连同文件一起缓存。

计费上是意外之喜：这个 CDN 和之前存储用的那个域名共用同一个流量包，没有产生新的收费项目，七牛的流量资源包对账户下所有加速域名通用。

顺便，我还做了个后台监控页，把服务器负载、七牛存储和 CDN 用量、密码管理器的存活状态聚合在一屏。趋势图我是自己用 SVG 画的，没有引 ECharts。为了一屏几条小折线拉进来一个几百 KB 的图表库，不划算。这里也踩了个坑：我这台是腾讯云的轻量应用服务器，不是标准的云服务器，查用量得用轻量专用的接口，标准云服务器的接口查不到数据。

## 被爆破的那阵子

站上线之后，有天我随手翻了翻服务器的认证日志。数字让我坐直了。

```viz
brute-force-timeline
```

上面这张图是真实数据，从服务器的认证日志里按天统计出来的（横轴是连续的天数，不是具体日期）。你能看到几个尖峰，单日两千多次登录失败。绝大多数冲着 `root` 账户来，零星夹杂着试 `ubuntu`、`admin`、`vpn` 这些常见用户名。全量统计下来，光一个来自某机房的 IP 就试了六千三百次。

这是自动化扫段爆破，不是针对我的定点攻击，全网的服务器都在挨这个。但看到这个量级，我还是决定认真做一次加固。

排查下来，好消息是当时并没有被攻破：能成功登录的记录全是我自己用密钥、从我自己的 IP 进的，一次密码登录成功都没有。那几千次对 root 的爆破从原理上就不可能得手，因为 root 压根没设密码。

但排查也翻出一个我没料到的真问题。root 在系统里是「空密码」状态——注意，是空密码，不是锁定。而系统的 PAM 认证默认带一个 `nullok` 选项，意思是「空密码也算通过」。这两个凑在一起，等于任何一个已经登进来的低权限账户，不用输任何密码，一条 `su root` 就能直接拿到 root。我当场试了一下，真的，一秒变 root。

这条路 SSH 加固根本挡不住，因为它压根不走 SSH。它的现实威胁是这样的：我的后端服务是跑在公网上的，万一应用或者某个依赖出了个能执行任意代码的漏洞，攻击者就拿到了后端的运行身份，然后一条 `su root`，整台机器就没了。

于是我分几层收窄：

**第一层，关掉密码登录。** 改 SSH 配置，只允许密钥认证。这一步直接让爆破从「猜不中」变成「没得猜」，服务器根本不再提供密码这个选项。我用一个额外的配置文件覆盖主配置，改完先校验语法、再 reload（而不是 restart，保住当前连接不断），然后开一个新终端验证密钥能进、密码被拒，才敢松手。这种改 SSH 的操作，得始终留一条后路，万一锁死了还有云服务商的网页控制台兜底。

**第二层，封死那个空密码后门。** 把 root 和那个闲置的默认账户的密码都锁定。root 本来就不靠密码登录，锁掉只是关掉 `su` 那条空密码通道，毫无副作用。

**第三层，给后端建一个专用的服务账户。** 这是针对「应用被攻破」那条路的正解。我新建了一个既不能登录、也没有 sudo 权限的系统账户，让后端服务用它来跑，数据库文件的属主也换成它。这样即使应用哪天真被人执行了任意代码，攻击者拿到的也只是一个啥也干不了的低权限身份——不能提权、不能碰系统的其它部分，只能读写它自己那个目录。整机保住。

**顺手，装了 fail2ban。** 同一个 IP 短时间内失败几次就自动封一段时间，屡犯加倍。在只剩密钥登录的前提下，爆破本已无效，这一步主要是把日志噪音和无谓的连接消耗压下去。

做完这几层，我心里踏实多了。安全这东西的思路，从来不是「造一堵攻不破的墙」，而是「就算某一层被穿了，损失也被关在一个小盒子里」。

## 顺便，那几张能点的图是怎么来的

你往上翻会看到三张图：一张能切场景的架构图、一张能播放的上传时序、一张能悬停读数的爆破柱状图。它们不是截图，是真的能点的组件。这是我给这个博客做的一个我自己挺得意的功能，值得单说两句。

Markdown 里想嵌个图，大多数人用 mermaid。我一开始也用，但很快嫌它憋屈：mermaid 画来画去就是流程图、时序图那几种固定样式，给你一套它的语法，你只能在它划的框里画。我想要的是能点、能拖、能播放的东西，那得是真正的交互组件，不是一张画死的图。

所以我干脆让文章能内嵌任意的 Vue 组件。写法学的就是 mermaid：正文里放一个 ```viz 加一个标识的围栏，渲染的时候，这段围栏会被替换成一个真正挂载上去的 Vue 应用。区别只在于，mermaid 围栏里是它的画图语法，而我这个围栏里，挂的是一个活的、有自己状态和交互的组件。

后台有个专门的编辑器管这些组件。左边一个框贴 Vue 单文件组件的源码，右边实时预览你写的东西长什么样、点起来什么反应。保存的时候有个关键动作：浏览器就地把这段源码编译成 JS（用的是 Vue 自己的编译器），然后把源码、编译产物、还有从组件里抽出来的那点 scoped 样式，一起存进数据库。

读者这边就轻松了。你打开文章，Markdown 渲染器碰到那个 viz 围栏，按标识去数据库取那份**已经编译好**的产物，当场 new 一个小 Vue 应用挂到围栏的位置。读者的浏览器不用做任何编译——编译在我保存的时候就一次性做完了，存下来的是成品。所以你看这几张图加载得挺快，它们本质上就是几个巴掌大的独立 Vue 应用，和文章正文存在同一个库里。

有个边界我一开始就划死了：只有登录的我能创建这种组件，而且组件源码被限制成只能 `import 'vue'`，别的什么都不让引。毕竟这是「把一段会执行的代码嵌进页面」，得管好谁能嵌、能嵌什么，不然就是给自己开后门。

写这篇文章时那三张图，就是这么来的。某种意义上，这篇讲怎么搭站的文章，正在用它自己搭出来的一个功能，讲它自己。

## 回头看

断断续续做下来，攒了一百多个提交。现在回头，有些账算得清：

赚得最明显的是 SQLite 和前端直传。SQLite 让我整个运维少了一大块心智负担：没有数据库进程要照顾，备份就是拷个文件。前端直传七牛让我这台小机器的带宽不用为文件传输买单，是它能扛住网盘功能的前提。

纠结过、也推翻过的是文件管理器。从现成的 VueFinder 换成自建，当时是个不小的决定，多花了成倍的时间。但现在每次用到那些贴合我习惯的交互，我都觉得那次重写没白干。经验是：现成组件适合探路，一旦你对细节较真，自己写反而更省心。

留着技术债的是不上数据库迁移工具。改一次模型就得重建库，这个坑我早晚要填，只是现在数据还不金贵，拖得起。

成本上，这套东西一年的开销小得让人意外：一台轻量服务器，一个域名，七牛的存储和 CDN 在免费额度加一个小流量包里打转。加起来是一顿饭钱的量级。

要说最大的收获，可能不是这个站本身，而是那种「把一个模糊的想法，一层层落成能跑、能扛、能上线的东西」的完整体验。从「我想要个带网盘的博客」到此刻你正在读的这个页面，中间每一个「为什么这么选」，我都亲手回答过一遍。慢是慢了点，但每一块我都清楚它为什么在那儿。
