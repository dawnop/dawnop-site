前两篇都在造语言：先有个能编译到 JVM 字节码、还能 native 编译的小语言，然后给它装上 typeclass。造语言的人有个通病——样例都是自己挑的，挑的都是语言擅长的。真要知道一门语言行不行，得拿它去干一件你本来就要干、且不许失败的活。

我手边正好有这么件活：你现在读的这个博客，后端是 FastAPI 写的，跑在生产上，带文章管理、七牛网盘、全文搜索、监控页。M6 的任务就一句话：**把它用 Dawn 重写一遍。**

不是新起一个玩具项目，是把这个正在服务的后端换掉。

## 绞杀者，不是掀桌子

一次性重写、然后某天切换，是重写工程里最经典的翻车姿势。所以从第一天就定死了策略：**绞杀者模式（strangler fig）**。

Dawn 后端起在 `127.0.0.1:8001`，FastAPI 继续在 `:8000`，两个进程**共用同一个 SQLite 库文件**（WAL 模式让它俩同时读写不打架）、共用同一个七牛空间。nginx 按 `location` 一条一条把 `/api` 的路由从 Python 切到 Dawn，切一条验一条，随时能切回去。回滚就是把一行 `proxy_pass` 指回 `:8000`、`nginx -s reload`——因为两端共享存储，没有任何数据迁移要撤。

```nginx
# fm 的代理上传（二进制请求体，最后迁）暂留 Python
location = /api/fm/upload { proxy_pass http://127.0.0.1:8000; }
# 其余 /api 全部走 Dawn
location /api/ { proxy_pass http://127.0.0.1:8001; }
```

这套的好处是心理上的：任何一刀翻了，影响面就是那一条路由，`reload` 一下就回去了，不用半夜回滚整个部署。

## 判据只有一个：逐字节对得上

「重写对了」怎么定义？我不信「看起来能跑」。判据定得很死：**同一个数据库副本、同一个 `SECRET_KEY`，两个后端对同一个请求的 JSON 响应，逐字段 diff，必须一致。**

这条纪律逼出了一堆细节。JWT 得跟 Python 的 PyJWT 双向互认——Dawn 签的 token FastAPI 认、反之亦然，否则切流的一瞬间所有登录态失效。密码哈希更刁：Python 的 bcrypt 存的是 `$2b$` 前缀，我用的 Java 库 jBCrypt 0.4 只认 `$2a$`——同一个算法，前缀不同，得在校验前归一化，否则每个老用户都登不进来。七牛的三类签名、腾讯云的 TC3-HMAC-SHA256，全部拿官方 SDK 生成参考值、逐字节对拍才敢用。

副作用端点没法直接 diff（删一个七牛对象是不可逆的），就换个法子：用官方 SDK 往一个 `dawntest/` 前缀上传隔离对象，跑 Dawn 的端点去操作它，再用 Python 复核结果、清理现场。生产文件的真实 key 一个都不进测试库。

## 语言几乎没长新语法

最出乎意料的一点：这么大一个真实后端，Dawn 的**语法几乎没动**。

这不是运气，是 M6 之前就定下的赌注（设计记录里叫 D8）：**语言不加 async、不加 web 框架，HTTP/JSON/SQL/加密全靠 `use java` 借 JVM 的现成轮子。** HTTP 用 JDK 自带的 `java.net.http` 加虚拟线程（每请求一线程，语言层零着色，没有 async/await 的颜色污染）；SQLite 走 JDBC 薄包装；JWT、bcrypt、七牛签名调现成 Java 库；JSON 用第二篇之前就写好的纯 Dawn 库。

于是 sql、crypto、http、web、json 这五套地基，全是拿 `use java` 加纯 Dawn 拼出来的，**留在应用层，一行都没进语言的标准库**。这恰好印证了那个判断：小语言不该自己长一套 web 栈，它只要把「借生态」这件事做顺就行。

真正逼出来的东西，全在**互操作的逃生门**上，而且都是不加语法负担的内建：

- `--cp`——上第三方 jar（sqlite-jdbc、jBCrypt），编译期反射和运行期加载共用一份 classpath。
- `java_try`——Dawn 没有异常，但 Java 世界的「预期失败」（网络断、SQL 约束冲突）是用异常表达的。这个内建是唯一的转换点，把异常收进 `Result`，而让真正的 bug（panic）照常穿透。
- `catch_panic`——监督边界。单个请求 panic 应该变成 500 记一笔，而不是掀翻整个进程。
- 二进制体零拷贝——取七牛对象的字节再透传出去，不需要把字节读进 Dawn 的值。擦除泛型返回的那个不透明 `Object`（实为 `byte[]`），在跨边界喂给 `OutputStream.write` 时做一次运行期 `CHECKCAST` 认领回来。大文件不顶爆内存。

一门语言好不好用，很多时候不看它有什么，看它缺的时候留没留好开口子。

## 一个 42 变成 42.0 的下午

对拍到监控页时，diff 报了个差异：七牛空间用量，FastAPI 输出 `51088624`，Dawn 输出 `51088624.0`。一个字节数，凭空长了个小数点。

追下去是这样：监控页的第三方数据带 TTL 缓存（避免每次刷新都去打七牛和腾讯的 API），缓存实现是把 JSON 响应 `render` 成字符串存起来，命中时再 `parse` 回来。问题就出在这个 round-trip：我的 JSON 解析器把**所有**数字都解析成浮点，`42` 进去、`42.0` 出来。第一次请求（没命中缓存）是对的，一旦走缓存就露馅。

修法不是在缓存那儿打补丁，是回到根上：让解析器对不带小数点和指数的字面量解析成整数。这是个正确的 JSON 库本该有的行为——一个把 `42` 读成浮点再吐成 `42.0` 的解析器，本身就是有损的。改完顺手检查了所有「从 JSON 读整数」的地方，发现绝大多数早就同时认整数和浮点两种形态，只有 JWT 的过期时间那一处需要补一笔。

真实项目的价值就在这种地方：这个 bug 在任何自选的样例里都不会出现，非得是「缓存 + 序列化往返 + 字节级对拍」三件事撞在一起才炸出来。

## 还没做完的那部分，我不打算含糊

代码层面，整个后端跑通了：公开站、全套后台内容管理、搜索、认证、文件管理、监控，逐字段跟 FastAPI 对得上。做了一次路由表全量对拍，结论是 **Dawn 覆盖了 FastAPI 的每一个 `/api` 端点，只差一个** `POST /api/fm/upload`（后端代理上传，multipart 二进制请求体）——它和 WebDAV 一样卡在同一个还没做的能力上：让 web 框架把请求体的原始字节交给 handler。这俩都是非浏览器的备份路径（前端上传走的是直传七牛、不经后端），一并推到 M6.5。

剩下的是纯运维动作，也是唯一没法在一个会话里做完的部分：把 jar 部署上生产、按 runbook 一段段翻 nginx、退役 uvicorn、然后**观察七天**——看有没有异常 5xx、内存稳不稳。一周的墙上时钟，急不来。

## 账面

十三刀，backend-dawn 约四千行 Dawn，四十六个单测全绿，路由 100% 覆盖（除掉那个明确推迟的）。语言这边新增的全是互操作逃生门，没加一条业务语法，spec 相应补了几节。

M6 想验证的其实是一个假设：**一门刻意保持小的语言，靠把「借 JVM 生态」做顺，够不够扛一个真实生产项目。** 现在能给的答案是够——而且它逼我补的东西，没有一样是「再给语言加个大特性」，全是「在边界上留个能借力的口子」。对一门一个人维护的语言来说，这是个让人踏实的结论。

下一步是把它真正切上生产、盯满七天。再下一步，是那件一直排在最后、也最要命的事：**让 Dawn 编译 Dawn**。
