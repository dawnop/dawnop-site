今年我给自己挖了个新坑：造一门语言。

动机不是发明什么新范式。类型系统、效果系统、模式匹配，这些东西学术界二十年前就想清楚了，工业界也各有各的成熟实现。我想验证的是一个更朴素的问题：2026 年，一个人用业余时间，能不能把这些「想清楚了的东西」组装成一门真的能用的语言？不是玩具解释器，是有类型检查、有像样报错、能编译出独立二进制、带 LSP 的那种。

所以预算先定死：编译器 6–8 千行 Kotlin，一年内做到能自举日常小工具。所有设计决策都从这个预算倒推。语言叫 Dawn。

先看一眼它长什么样：

```rust
type Shape =
  | Circle(r: Float)
  | Rect(w: Float, h: Float)
  | Point

fn area(s: Shape) -> Float =
  match s {
    Circle(r)  -> 3.14159 * r * r
    Rect(w, h) -> w * h
    Point      -> 0.0
  }

pub fn main() -> Unit !io = {
  let shapes = [Circle(1.0), Rect(2.0, 3.0), Point, Circle(200.0)]

  shapes
    |> map(area)
    |> fold(0.0, fn(acc, a) => acc + a)
    |> fn(total) => println("total area: {total}")
}

test "area of unit circle" {
  assert area(Circle(1.0)) == 3.14159
}
```

这段代码不是宣传稿里的伪代码。它是设计第一天写进 `examples/shapes.dawn` 的验收样例，当时一行编译器都还没有。现在，它一个字没改，通过了 `dawn run` 和 `dawn test`。后面会讲为什么这句话值得强调。

## 一个决策省掉半个编译器

我对这门语言有两个目标形态的要求：要能吃到 JVM 的生态和 GC，也要能出独立的 native 二进制（写命令行小工具，没人想等 JVM 起）。常规答案是两个后端，那是团队的活，一个人碰都别碰。

Dawn 的答案：**唯一后端是 JVM 字节码**，Kotlin + ASM 直出，连中间 IR 都没有（AST 上标注类型，然后直接出字节码，优化交给 JIT）。native 不是第二个后端，而是 GraalVM native-image 把同一份字节码再 AOT 一遍。`dawn build --native` 干的事，只是在产出 jar 之后多接一条 `native-image` 命令。

这个便宜不是白捡的。native-image 有封闭世界假设：它静态分析可达代码，反射、动态类加载、`eval` 都是雷，正常项目要靠一堆 JSON 配置文件去声明「运行时会摸到哪些类」。Dawn 的做法不是绕，而是**语言设计上根本不提供这些特性**：没有反射，没有运行时加载代码，方法分发全是 `invokestatic` / `invokevirtual`，闭包用 `LambdaMetafactory`（它在 native-image 的官方支持名单里）。这些能力本来也不在一门小语言的目标里，于是 native 编译不需要任何配置文件，不是「碰巧不需要」，是 by construction 不需要。

整条管线摊开是这样，点每一步能看到同一个函数（就是上面的 `area`）在那一步的真实形态，字节码那格是 `javap` 的真实输出：


## 签名必须说真话

如果说上面是工程决策，这一节是 Dawn 真正的立身特性：**纯度是类型系统的一部分**。

函数默认是纯的。想碰 IO（打印、读文件、网络），签名末尾必须标 `!io`，而且这个标记是传染的：调了 `!io` 函数的函数，自己也得标。编译器推导正文的实际效果，再和签名声明的对账，对不上就不放行。于是看签名就知道一个函数会不会碰外界，这不靠约定和自觉，靠编译器。纯函数的甜头到处都是：测试不用 mock 任何东西，将来 comptime 也只允许调纯函数。

为什么不上完整的代数效应（effect handlers）？因为那是工程黑洞：类型推导、错误信息、教学成本全部翻倍，正是一个人预算里最先要砍的东西。Dawn 的效果格子只有两个点，pure 和 io，实现上不过是检查器里多传一个格子值。

但有一个问题两级格子躲不掉：高阶函数。`map` 自己不碰 IO，它纯不纯完全取决于你传进来的函数。把 `map` 标成 `!io` 太冤（传个纯函数进去也被传染），不标又是谎言。答案是最小剂量的效果多态，**效果变量**：

```rust
fn map[T, U](xs: List[T], f: fn(T) -> U !e) -> List[U] !e
```

`!e` 读作「`map` 的效果 = `f` 的效果」，在每个调用点单独实例化。下面这张图就是这件事本身，四种组合都拨一遍，右下角的报错是编译器的真实输出：

```viz

```

## 穷尽性检查不能瞎保守

ADT 加 `match` 是这门语言的骨架，而 `match` 必须穷尽：漏一个构造器就是编译错误。这个检查的难点在「不多不少」，多报比漏报更伤，用户会立刻学会无脑加 `_ ->` 兜底，检查就形同虚设了。

我的第一版实现就栽在这上面。朴素思路是记账：看看每个类型的构造器是不是都出现过。平铺的 match 没问题，一碰嵌套就误报，比如对 `Option[Option[Int]]` 写 `Some(Some(v))` / `Some(None)` / `None` 三个分支，明明已经滴水不漏，记账法却看不出来，因为「都出现过」不等于「组合覆盖全」。

这个问题的正解业界早有定论：Maranget 2007 年的 **usefulness 算法**。核心是把问题反过来问，不问「覆盖了没有」，问「一个分支有没有用」：

> 一个分支「有用」，当且仅当存在某个值，前面的分支都接不住、它能接住。
> match 穷尽，当且仅当在所有分支后面补一个 `_`，这个 `_` 没用。

一个谓词，递归地在模式矩阵上做构造器特化，同时解决了三件事：穷尽性、冗余分支检测（没用的分支就是永远匹配不到的分支）、缺失报告（探测哪些构造器让补出来的 `_` 有用）。第一版那些误报，换上它之后一次全消。

## 泛型走了最无聊的路

泛型的实现路线选了最没有惊喜的一条：**擦除加装箱**。类型参数 `T` 擦成 `Object`，`Int` 经过泛型位置时装箱成 `Long`。不做 C++ / Rust 式单态化（代码膨胀，还和 native-image 的封闭世界打架），不做 Kotlin 的 reified。慢一点，但行为可预测，且 JVM 的 JIT 大部分时候能把装箱开销救回来。逃生门也留了：装箱只发生在顶层类型变量的位置，`fn add(a: Int, b: Int) -> Int` 这种具体签名从头到尾都是原生 `long`。

有意思的部分在类型推导。Dawn 没有全局 Hindley-Milner，只做**调用点合一**：拿实参类型和形参类型对齐，解出类型变量。这个选择本身也是预算驱动的（HM 的错误信息出了名的难做好），但两个小设计让它够用了：

- **期望类型播种**：`let x: Option[Option[Int]] = Some(None)`，先用期望类型把外层 `Some` 的 `T` 绑成 `Option[Int]`，再去检查实参，于是里面那个光杆 `None` 也有了着落。
- **两轮实参检查**：`or_default(None, 5)` 里，`None` 自己推不出 `T`，但兄弟实参 `5` 可以。所以第一轮先走「自足」的实参把类型变量绑上，第二轮再回头检查 `None`、空列表、未标注参数的 lambda 这些「需要上下文」的家伙。

两条规则都很土，但覆盖了日常代码里几乎全部的推导场景，而且错误落点精准：推不出来时，报错指着那个具体的实参，而不是整个调用。

## 免配置差点破功

前面吹了 native 编译零配置是 by construction，这里得坦白：它差点在闭包上破功。

Dawn 的 lambda 走 JVM 标准路径 `LambdaMetafactory`（Java 自己的 lambda 也是它），选它就是因为它在 native-image 的支持名单里。但第一次给 lambda 跑测试，还没到 native，JVM 上就炸了：

```text
LambdaConversionException: Type mismatch for instantiated parameter 0:
  class java.lang.Object is not convertible to long
```

排查下来是一个不读规范就会踩的坑：`LambdaMetafactory` 要两份方法签名，一份是擦除的接口签名，一份是「实例化签名」（instantiatedMethodType）。我图省事两份都传了擦除版，全是 `Object`。而 LMF 对适配是有规矩的：`Object → long` 不是合法适配（那是拆箱加窄化两步），`Long → long` 才是。修法就是老老实实把实例化签名写成装箱后的具体类型；顺带牵出第二个坑，LMF 不适配 `void`，返回 `Unit` 的 lambda 实现方法必须返回 `Object`（一个 `null`），为此还得给作为值传递的 `Unit` 函数生成桥方法。

修完在 native-image 下实测：lambda、高阶函数、效果变量全套跑通，依旧一个配置文件都没有。承诺保住了，但这一段的教训比结论值钱：**「在支持名单里」和「你用对了」是两回事**。

## 语言自带测试

最后一块拼图是测试。Dawn 的测试不是框架，是语法：

```rust
test "point has no area" {
  assert area(Point) == 0.0
}
```

`test` 块写在源文件里，和被测代码住一起。`dawn test` 编译并逐个执行，`dawn build` 则直接剥除，产物里不带一字节测试代码。`assert a == b` 失败时不是干巴巴一句 assertion failed：编译器把 `==` 拆开、两侧各存进临时变量，失败时连源码文本带两侧的值一起报出来。这是从「纯函数测试零 mock」顺下来的设计：测试便宜到没有借口不写。

错误处理也顺便说一句。Dawn 没有异常（异常是另一种签名谎言），可恢复错误走 `Result[T, E]` 加 `?` 传播。`?` 的编译产物我很喜欢：失败分支直接把手里那个 `Err` / `None` 实例 `areturn` 返回出去，因为擦除之后它的类就是声明的返回类型，一次分配、一条多余的字节码都没有。

于是回到开头那句话。`shapes.dawn` 是设计第一天写下的验收样例：ADT、嵌套模式、守卫、管道、lambda、效果标记、test 块，当时全是空头支票。九个 commit 之后，它一个字没改跑通了，JVM 和 native 二进制的输出逐字节一致。用一个固定样例当里程碑的验收线，是这个项目里最值的一个流程决策：它逼着每一刀都朝同一个方向切，也让「做完了」有一个不容自欺的定义。

## 现状与下一步

账面数字：编译器 4900 行 Kotlin（预算 6–8 千行，还有富余），测试 114 项，native 二进制启动约 7 毫秒。LSP 从 M0 起就一路同步演进，诊断、悬停、跳转、大纲都有，前端做了完整的错误恢复，文件写残了其余部分照常分析。

下一个里程碑（M2）是三个立身特性里还欠的两个：**comptime**（编译期跑纯函数，产物烧进常量池，没有宏）和 **Java 互操作**（`use java` 直接调 Java 类，返回值自动包 `Option`，null 进不了语言），外加字符串处理这类标准库地基。验收样例照旧提前写好了，`examples/calc.dawn` 正躺在仓库里等着编译失败。
