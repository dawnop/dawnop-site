上一篇的结尾写着：再下一步，是那件一直排在最后、也最要命的事——让 Dawn 编译 Dawn。

为什么最要命。编译器大概是一门语言能写的最难的一类程序：递归数据结构层层嵌套、状态多到溢出、性能敏感、错一个字节就全盘皆输。一门语言如果写不了自己的编译器，说明设计有洞；如果写得了、但得先给自己开几个后门，说明设计破产。自举是语言设计的终审。

它还有个现实的代价：自举之后，语言每改一处都要同时伺候两个编译器。所以它必须排在语言基本冻结之后——这也是它一直排在最后的原因。

## 判据只有一个，还是逐字节

「重写对了」怎么定义？上一篇重写后端时的纪律是两个后端的 JSON 响应逐字段 diff。这次的纪律更硬：**编译器的每一层输出，与 Kotlin 版逐字节相同。**

Kotlin 版开了四个隐藏命令——`__lex`、`__parse`、`__check`、`__emit`——把词法、语法、类型检查、代码生成四层的中间结果各自 dump 成规范格式；Dawn 版实现同样四层，落同样的格式。对拍语料是仓库里**每一个** `.dawn` 文件：所有示例、站点生成器（发射出 99 个类）、playground 后端（91 个类），最后加上编译器自己。四个命令、四份语料两边各跑一遍，`diff -r` 逐字节比。任何一个字节不同，CI 红灯。

选字节而不是「行为一致」，理由和上次一样：行为等价是测不完的，字节相等是一条 `diff -r` 就能验的全称命题。而且它会复利——词法层逐字节相同，语法层的输入就严格相同，出了差异永远只用查当前这一层。四层就是四道闸门，差异被夹在它第一次出现的那一层里，不会漏到下游变成一桩悬案。

## 把一个可变的编译器翻译成不可变的

真正的工作量不在「把算法翻译对」。算法是公开的、两边一样。难的是：Kotlin 版编译器的骨架，恰好由三样 Dawn **一样都没有**的东西撑着——可变字段、循环引用、靠对象指针分辨身份。要把它搬进 Dawn，这三样都得换一套活法，而且换完还要发射出一模一样的字节。

**身份不能再靠指针。** 类型检查器里，两个同名的类型变量 `T` 必须能区分——一个函数的 `T` 不是另一个函数的 `T`。Kotlin 靠对象身份：`new TVar("T")` 两次就是两个不相等的对象，`===` 天然分开。Dawn 的记录是值，两个 `TyVar("T")` 结构相等、无从分辨。解法是给每个类型变量发一个整数 id，相等按 id 比——身份从「你是哪个指针」翻译成「你是几号」。效果变量同理。

更硬的是循环引用。类型信息和它的构造器互指：一个 ADT 知道自己有哪些构造器，每个构造器又指回自己属于哪个 ADT。Kotlin 让两个可变对象互相持有引用就成环了；Dawn 的记录不可变，**没法成环**——你要构造 A 就得先有 B，要构造 B 又得先有 A。解法是把「指针」再降一级成 id：所有 ADT 存进一张 `Map[Int, AdtInfo]` 表，构造器只存 `adt_id: Int`，谁要顺着走就查表。递归类型（`List[List[Int]]` 这种自我指涉的）还要分两遍建：第一遍只注册空壳（id、名字、类型参数），第二遍才填字段——因为填字段时可能要引用还没建完的自己。指针能一步到位的地方，值语义得拆成两步。

**可变状态收进一个记录里穿。** Kotlin 的检查器是一大堆可变字段：诊断列表、作用域栈、fresh 计数器、当前效果上下文……代码生成器更甚，寄存器槽计数、待发射的 lambda 队列、常量池全是可变的。Dawn 没有可变集合，也不打算为编译器破例——那等于承认这套设计写不了正经程序，自举就没意义了。做法是把所有可变字段收进一个记录，像传参一样线程化：检查器全程穿一个 `Cx`，代码生成器全程穿一个 `Gen`，每个函数收一个、返回一个新的，谁改了什么全写在签名里。账面代价是行数——Dawn 版比 Kotlin 版长了五成；换来的是整个编译器没有一处隐式状态，调试时这五成又赚回来了。

**连异常都没有。** 编译期常量求值是个小解释器，Kotlin 版用异常做控制流：提前 return、break、continue、报错，全是往上抛。Dawn 没有异常。于是这套控制流被翻译成一个 `Result` 加一个专门的和类型 `Ctl`（分 return / break / continue / 错误四种），Dawn 的 `?` 运算符就当 throw 用——函数调用捕 return，循环捕 break/continue。异常的「非局部跳出」被显式地编码成了值的传递。有个副作用挺好玩：这个编译期解释器在宿主栈上递归，跑深了会爆栈，所以运行 Dawn 版编译器得给 JVM 加 `-Xss512m`——一个自己写的编译器，因为它自带的求值器递归太猛，得先把栈开大才编得动自己。

**检查结果没处写回。** Kotlin 的类型检查器边查边往语法树节点上写：这个表达式是什么类型、这个调用点消解到哪个实现、这个循环里有没有 break——四十来个可变槽，直接钉在树节点上。Dawn 的语法树是不可变的记录，写不回去。于是检查器不改树，而是**长出另一棵树**：一棵带类型标注的树（内部叫 TAST），每个节点自带它的类型，和检查阶段定下的全部结论。往下走的编译期解释器和代码生成器都读这棵树，谁也不回头改语法树。这棵树上最有意思的一类标注，是 trait 的见证——它值得单开一节。

这一节的每一处翻译，Kotlin 一个指针、一个 `throw`、一次写回就了事的地方，Dawn 都要落成一张表、一个 id、一棵新树、一个显式传递的值。慢、啰嗦，但每一处状态流动都摊在明面上。这就是「不开后门」的实际售价。

## 身份翻译的报复

把指针身份换成整数 id，听着直接，代价藏在一个地方：**id 得有人来发，而发重了两个不同的东西就成了同一个。** 指针不会撞，计数器会。

第一次是内建类型撞车。用户定义的第一个类型分到 id 1，而内建的 `Result` 恰好也占着 1——于是任何程序只要自己定义了个类型，`Result` 就被顶掉了。第二次更隐蔽：跨模块编译时，每个模块的 id 各自从 2 起，一个模块把类型嫁接到另一个模块里，两套「2 号」撞上，playground 全线崩。指针世界里根本不存在的这类 bug，是把身份降维成整数之后凭空长出来的。修法是让 id 计数器从标准库一路串到每个用户模块，全局只有一个源头在发号，谁都不重复。

这是自举给的一个具体教训：你不是在「翻译」一个编译器，是在一套完全不同的世界观（值、不可变、无异常）里**重新实现**它。两套世界观的接缝处，会长出原版从没有过的 bug。

## 见证消解：把一个决定冻进树里

第二篇讲 trait 时，说过实现侧的主角是「见证消解」：`a < b` 里的 `<` 到底调谁，检查器在每个调用点定死——具体类型去虚化成一条直呼，类型参数则转发一份隐藏字典。那篇讲的是这套机制**是什么**；自举这篇要补的是它在两版编译器之间**以什么形态存在**。

Kotlin 把消解结果写在调用节点的一个可变槽里（`e.witnesses`）。Dawn 写不回节点，于是这个结果被**冻进上一节那棵 TAST**——见证在树里是一个和类型，只有两种形态：

- **`WConcrete(trait_id, subject)`**——调用点的类型是具体的（`T = Card`）。一致性保证「`Ord` 给 `Card`」的实现全程序唯一，所以树里只需记下 trait 和主体这两个坐标；代码生成时拿它去查 impl 表得到那唯一实现，发一条 `invokestatic` 直呼，零动态分发。
- **`WForward(sym)`**——调用点的类型还是个类型参数（泛型函数调泛型函数）。树里记下「转发调用者的第几个字典参数」；代码生成时 `ALOAD` 那个槽、`invokeinterface`。这是全系统唯一残留的动态分发。

注意那个 impl 表：它的键是 `(trait_id, subject)`，而 `subject` 是个类型——**又回到了「身份换成 id、循环换成表」那套**。类型能直接当哈希键，是因为能进 impl 主体的类型全是 ADT，没有 `Float`/`Bytes` 这类相等语义暧昧的东西。连「这个约束用哪个实现」这么一个决定，最后都落成一次查表。

移植里最省心的一处，还是第二篇那个决定在替我省事：**字典就是一个合成的局部变量**（一个 Symbol，发个整数 id，擦除后占一个槽）。它带来的好处在 lambda 上兑现——`fold(xs, first, fn(acc, x) => if acc < x { x } else { acc })`，lambda 体里的 `<` 要用外层函数的字典，而 TAST 里 lambda 节点的捕获列表存的就是一串 Symbol id，字典 Symbol 跟普通局部变量在树里长得**一模一样**。于是捕获分析原样把它捎进闭包，移植代码生成器时，这里针对字典的特判一行都没写。为「字典」发明一个专门表示、再到处开洞，是移植时同样能踩的坑——这个决定在第二篇省一次，自举时又省一次。

但字节级一致又逼出一个新要求，是概念层面看不见的：**槽的顺序**。隐藏字典参数排在声明参数后面，多个约束按固定次序占槽——这个次序（TAST 里记作 `dict_syms`，就是检查器绑定字典的先后）两版必须逐一对齐。错一个，字典的槽号就错，`ALOAD` 取错、字节码验证当场不过。函数描述符也跟着这个次序走：一个受 `[T: Ord + Describe]` 约束的函数，擦除后的签名是「声明参数 + 两个 `Object`」，这两个 `Object` 谁先谁后，正是 `dict_syms` 的先后。概念上「多传一个字典」，落到字节上是「第几号槽、第几个 `Object` 参数」，一位都不能差。

两头的极端也都得照抄字节：标量最省，`cmp(1, 2)` 里的 `Ord` 见证根本不查表，直接内联成一条 `LCMP`；`derive Ord` 生成的字段字典序 `cmp`（记录逐字段比、和类型先比构造器序）挂在同一张 impl 表上，对见证消解来说和手写 impl 没有任何区别。这些形态在第二篇都定了型，自举要做的，是把每一种情形——去虚化的直呼、转发的字典、内联的标量、derive 的字段序——发出的字节，一条不差地再复现一遍。

## 一致到什么程度才算一致

翻译对了逻辑，还有一层更阴的东西：那些**不在源码里、却决定字节**的顺序和默认值。

最典型的是栈映射帧。JVM 要求每个方法带一张「帧表」，由 ASM 的 `COMPUTE_FRAMES` 自动算，算的时候要解析「两个类型的公共超类」——可生成中的类还不在 classpath 上，ASM 查不到，得靠编译器自备一张类层级表回答。这个解析两边必须是**同一个实现**，否则同一段代码两边算出的帧不同，字节当场岔开。所以整个工程里唯一两边共用的代码，是一个百来行的 Java 垫片：一个覆写了「查公共超类」的帧写入器。它不属于任何一边，是字节级一致的支点，随种子逐代续传、从不重编。

其余全是顺序，而且每一处都被金样钉死：lambda 先于各类桥方法发射、桥方法内部还分先后（函数值桥、内建桥、构造器桥、SAM 桥，按固定的四步顺序排空队列）、常量池按首次使用序物化、一个类里方法的排列次序……任何一处抄错顺序，字节就红一片。这些顺序在 Kotlin 版里是「代码碰巧这么写的」，在 Dawn 版里必须是「明知故犯地照抄」。

最阴的一个是装箱。Dawn 的 `Int` 是 64 位，跨过互操作边界喂给 ASM 时会装成 `java.lang.Long`，于是发射出来的是长整型常量；而字节码里很多位置要的是 32 位 int 常量，得显式装成 `Integer` 才对。同一个 42，装成 `Long` 是一条指令，装成 `Integer` 是另一条。金样当场标红，找了一下午——因为源码看上去完全正确，错的是一个你根本没写出来、由互操作边界替你做的隐式装箱。

## 只有「自己编自己」才炸得出来的 bug

全部语料绿灯之后，让它第一次编译自己——panic：某个跨模块常量在编译期没有值。

原因挺深：常量的编译期求值结果，存放在**声明它的模块**的输出里，引用方只带一个名字。语料里的程序都太「正常」，没有谁像编译器自己这样，几十个模块互相引用彼此的编译期常量。第一个把这条路走出来的程序，就是它自己。修法是在总装时按「名字 → 声明模块」把各模块的常量表拼起来，引用方顺着名字去对的那张表里取值。

修完，stage2 出来了：Dawn 版编译器发射自己的全部 433 个类，与 Kotlin 版发射的逐字节一致。再用这批**自己发射的类**跑同一次发射——又炸一个：vendored 的运行时类之前总是从 jar 资源里拷，而这次编译器是从一个类**目录**跑起来的，资源协议从 `jar:` 变成了 `file:`，拷贝那步找不到文件。补上对 `file:` 协议的支持，过。

两个 bug 共一个教训：语料覆盖得再全，也替代不了「自己编自己」这一步。编译器是唯一一个和它本身一样大、一样怪的测试输入——几十个模块、密集的跨模块引用、从类目录而非 jar 启动，这些角落只有它自己会去踩。

## 固定点，然后闭包

修完这两处，三段链走通：

- **stage1**——Kotlin 版编译 Dawn 版，得到一个 Dawn 写的编译器（跑在 JVM 上）。
- **stage2**——用这个 Dawn 版编译器编译它自己的源码，发射出 433 个类。
- **stage3**——再用 stage2 刚发射的那批类，编译同一份源码。

**stage2 与 stage3 逐字节相同。** 这就是教科书里的自举固定点：一个编译器发射出的自己，再拿去发射自己，结果不动了。到这一步，Kotlin 已经从链条里消失——stage3 的产物只由 Dawn 源码决定，跟当初是谁把它引导出来的无关。

```viz
dawn-selfhost-chain
```

但还差最后一步才算真正独立：那个帧写入器垫片和 ASM 本身，还住在 Kotlin 编译器的 jar 里，跑 Dawn 版得把那个 jar 挂在 classpath 上。于是补了 `build` 模式：把程序类连同这两样按包名拷进产物，用一个确定性的 jar 写入器落盘——manifest 在前、每个条目的时间戳钉死，同一份类表必然产出同一串字节（jar 里但凡有一个时间戳跟着墙上时钟走，两次构建就 cmp 不过）。

终局验收两条，都在 CI 里，每次 push 都要过：**独立**——classpath 上只挂这一个 jar，重新发射语料，与参照逐字节一致；**闭包**——用这个 jar 重建它自己，两个 jar 文件 `cmp` 相同。一个文件生成一个与自己逐字节相同的文件，这句话在别处近乎玄学，在这儿是绿灯的日常。

## 冻结，以及冻结之后怎么活

Go 在 1.5 完成自举后，把最后一个 C 写的版本 go1.4 永久保留为 bootstrap 种子。Dawn 照做：Kotlin 版随 v0.6.0 冻结，release 归档的那个 jar 就是种子。从零重建的链条从此固定：种子编译 Dawn 版 → Dawn 版发射自己到固定点 → 打出独立 jar → jar 重建自己收闭包。

冻结说得具体些：bug 修照收，种子一旦失去「能编译 Dawn 版」的能力，算最高级别故障；但新语言特性默认不再进 Kotlin 版——要做就得两边同做，这份双倍成本是刻意装上的刹车。也照例不含糊该含糊的：日常工具链（run、test、fmt、LSP）当时仍是 Kotlin 版在服务，把它们迁进 Dawn 版是后话，不是冻结的前提（这之后确实一件件迁完了，那是另一段故事）。

冻结带出一个新问题：Kotlin 那个「参照答案」退休之后，拿什么继续保证每次改动没把字节改坏？答案是把对拍从「跨实现」换成「跨版本」——**这一版编译上一版的语料，逐字节比**。上一个 release 和当前 HEAD 编同一批代码，字节不该变；真要变（比如故意改了发射策略），提交信息里得写明一行声明，否则 CI 红灯。参照系从「另一门语言写的编译器」变成「上一个自己」，自举之后这门语言就是这样自己给自己打分的。

## 账面

Dawn 版编译器 22 个文件、约 2.2 万行 Dawn，类型检查器 7.8 千行是大头，代码生成 3.6 千行，比 Kotlin 版长五成，那五成买的是「没有一处可变字段、没有一处隐式状态」。验收七道全部进 CI：四层金样、固定点、独立、闭包。有个连我自己都要核对两遍的事实：从词法器的第一刀到能重建自己的 jar，提交历史压在同一天里——当然，前面有两批清障（字符串游标、标准库重组）垫了路，路是提前修好的，那天只是把车开了过去。

系列第一篇造了语言，第二篇给它装上 typeclass，第三篇拿它换掉了这个博客的后端。这一篇之后，有件事起了质变：**这门语言的全部编译行为，从此由一个用它自己写的程序定义。** Kotlin 只剩一个身份——种子。

一门一个人维护的小语言，把「写不了正经程序」的嫌疑排除到这个程度，够本了。
