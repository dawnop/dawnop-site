上一篇的结尾写着：再下一步，是那件一直排在最后、也最要命的事——让 Dawn 编译 Dawn。

为什么最要命。编译器大概是一门语言能写的最难的一类程序：递归数据结构层层嵌套、状态多到溢出、性能敏感、错一个字节就全盘皆输。一门语言如果写不了自己的编译器，说明设计有洞；如果写得了、但得先给自己开几个后门，说明设计破产。自举是语言设计的终审。

它还有个现实的代价：自举之后，语言每改一处都要同时伺候两个编译器。所以它必须排在语言基本冻结之后——这也是它一直排在最后的原因。

## 判据只有一个，还是逐字节

「重写对了」怎么定义？上一篇重写后端时的纪律是两个后端的 JSON 响应逐字段 diff。这次的纪律更硬：**编译器的每一层输出，与 Kotlin 版逐字节相同。**

Kotlin 版开了四个隐藏命令，把词法、语法、类型检查、代码生成四层的中间结果各自 dump 成规范格式；Dawn 版实现同样四层。对拍语料是仓库里**每一个** `.dawn` 文件——所有示例、站点生成器（发射出 99 个类）、playground 后端（91 个类），最后加上编译器自己。任何一个字节不同，CI 红灯。

选字节而不是「行为一致」，理由和上次一样：行为等价是测不完的，字节相等是一条 `diff -r` 就能验的全称命题。而且它会复利——词法层逐字节相同，语法层的输入就严格相同，出了差异永远只用查当前这一层。四层就是四道闸门。

## 不给自己开后门

编译器爱可变状态：符号表、寄存器槽计数器、待发射的 lambda 队列，Kotlin 版的代码生成器里全是可变字段。而 Dawn 的内核是不可变数据加显式效果，没有可变集合。

最大的诱惑就在这儿：给编译器开个口子，加一个「内部用」的可变容器，一切照抄就行。没开。为写自己的编译器破例，等于承认这套设计写不了正经程序——那自举就没有意义了。

修法是把 Kotlin 的可变字段收进记录，像传参一样线程化：类型检查器全程穿一个 `Cx`，代码生成器全程穿一个 `Gen`，每个生成函数收一个 `Gen`、返回一个 `Gen`，谁改了什么在签名里一目了然。账面代价是行数：Dawn 版 22 个文件约 2.2 万行，比 Kotlin 版的 1.5 万行长了五成（其中也内联着 113 个单测）。换来的是整个编译器没有一处隐式状态——调试时这五成又赚回来了。

## 一致到什么程度才算一致

真正难的不是把逻辑翻译对，是那些**不在源码里、却决定字节**的东西。

JVM 的栈映射帧由 ASM 的 `COMPUTE_FRAMES` 计算，计算时要解析「两个类型的公共超类」——而生成中的类还不在 classpath 上，得靠编译器自备的类层级表。这个解析两边必须是**同一个实现**，否则同一段代码两边算出不同的帧、字节就岔开。所以整个工程里唯一两边共用的代码，是一个百来行的 Java 垫片：帧写入器。它是字节级一致的支点。

其余全是顺序：lambda 先于各类桥方法发射、桥方法之间还有先后、常量池按首次使用序物化——每一处顺序都被金样钉死，抄错一处红一片。

最阴的一个是装箱。Dawn 的 `Int` 是 64 位，跨过互操作边界喂给 ASM 的 `visitLdcInsn` 时会装成 `java.lang.Long`，于是发射出来的是长整型常量；而字节码里很多位置要的是 32 位 int 常量，得显式装成 `Integer` 才对。同一个 42，装错箱就是另一条指令。金样当场标红，找了一下午。

## 只有「自己编自己」才炸得出来的 bug

全部语料绿灯之后，让它第一次编译自己——panic：某个跨模块常量在编译期没有值。

原因挺深：常量的编译期求值结果存放在**声明它的模块**的输出里，引用方只带名字。语料里的程序都太「正常」，没有谁像编译器自己这样，几十个模块互相引用彼此的常量表。第一个踩出这条路的程序，就是它自己。

修完，stage2 出来了：Dawn 版编译器发射自己的全部 433 个类，与 Kotlin 版发射的逐字节一致。再用这批**自己发射的类**跑同一次发射——又炸了一个：vendored 运行时类之前总是从 jar 里拷，而这次编译器是从一个类目录跑起来的，资源协议变了。

两个 bug 共同的教训：语料覆盖得再全，也替代不了「自己编自己」这一步。它是唯一一个和编译器本身一样大、一样怪的测试输入。

## 固定点，然后闭包

修完这两处，三段链走通：stage1（Kotlin 版发射 Dawn 版）、stage2（Dawn 版发射自己）、stage3（用 stage2 的产物再发射一遍）——**三者逐字节相同**。这就是教科书里的自举固定点。

但还差最后一步才算独立：那个帧写入器垫片和 ASM 本身，还住在 Kotlin 编译器的 jar 里，跑 Dawn 版得把那个 jar 挂在 classpath 上。于是补了 `build` 模式：把程序类连同这两样按包拷进产物，用一个确定性的 jar 写入器落盘——manifest 在前、每个条目的时间戳钉死，同一份类表必然产出同一串字节。

终局验收两条，都在 CI 里：**独立**——classpath 上只挂这一个 jar，重新发射语料，与 Kotlin 版逐字节一致；**闭包**——用这个 jar 重建它自己，两个 jar 文件 `cmp` 相同。一个文件生成一个与自己逐字节相同的文件，这句话在别处近乎玄学，在这儿是每次 push 都要过的一步。

## 冻结

Go 在 1.5 完成自举后，把最后一个 C 写的版本 go1.4 永久保留为 bootstrap 种子。Dawn 照做：Kotlin 版随 v0.6.0 冻结，release 归档的 jar 就是种子。从零重建的链条从此固定：种子编译 Dawn 版 → Dawn 版发射自己到固定点 → 打出独立 jar → jar 重建自己收闭包。

冻结说得具体些：bug 修照收，种子失去「能编译 Dawn 版」的能力算最高级别故障；但新语言特性默认不再进 Kotlin 版——要做就得两边同做，这份双倍成本是刻意装上的刹车。也照例不含糊该含糊的：日常工具链（run、test、fmt、LSP）今天仍是 Kotlin 版在服务，把它们迁进 Dawn 版是后话，不是冻结的前提。

## 账面

Dawn 版编译器 22 个文件、约 2.2 万行 Dawn，类型检查器 7.8 千行是大头，代码生成 3.6 千行。验收七道全部进 CI：四层金样、固定点、独立、闭包。有个连我自己都要核对两遍的事实：从词法器的第一刀到能重建自己的 jar，提交历史压在同一天里——当然，前面有两批清障（游标、标准库重组）垫了路，路是提前修好的，那天只是把车开了过去。

系列第一篇造了语言，第二篇给它装上 typeclass，第三篇拿它换掉了这个博客的后端。这一篇之后，有件事起了质变：**这门语言的全部编译行为，从此由一个用它自己写的程序定义。** Kotlin 只剩一个身份——种子。

一门一个人维护的小语言，把「写不了正经程序」的嫌疑排除到这个程度，够本了。
